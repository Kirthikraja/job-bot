Resume ingestion — how it works (job-bot)
==========================================

Who should read this
--------------------
Anyone wiring uploads, RAG, or debugging “why is my resume not in Chroma?”
You do not need to already know LangChain; terms are defined below.


What “resume ingestion” means (plain)
--------------------------------------
After we parse a resume, we have a JSON file on disk (structured fields:
name, skills, jobs, etc.). “Ingestion” means: read that JSON, cut the text
into smaller pieces (“chunks”), turn each piece into a vector embedding, and
store those vectors in Chroma (a local vector database under
backend/ai/persists/). That index is used later when you want search-style
retrieval over resume text (chunk RAG). It is NOT the same as saving the JSON
file — ingestion is an extra step on top of the JSON.


Words you will see
------------------
- resume_id: A string that names one resume VERSION in your app (example:
  "pm-v1" or a UUID). You choose it in the API when handling upload — not
  inside the PDF and not invented by rag.py.
- Parsed JSON file: The output of save_parsed_resume — one file per
  resume_id when you use multi-resume paths (see below).
- Chunk: A substring of the JSON text (roughly up to 1000 characters) so
  embeddings stay within model limits and search can return a small relevant
  slice.
- Chroma: Vector store; each row has text + metadata + embedding.
- doc_id: A metadata field on every chunk. All chunks from the SAME resume
  version share the SAME doc_id value so we can delete or search “only this
  resume” without touching others.


End-to-end story (happy path)
-----------------------------
1) User uploads a PDF.
2) Backend saves the PDF somewhere (path is your choice per version).
3) resume_parser: PDF text → Gemini → dict → validate_parsed_resume.
4) save_parsed_resume MUST write that dict to the JSON path that ingestion
   will read. For multiple resumes that path is:
   uploads/parsed_resumes/<resume_id>.json
   (built by parsed_resume_json_path in rag.py — use the SAME resume_id for
   save and ingest).
5) ingest_resume(resume_id) in rag.py:
   - Builds the same path, loads the dict.
   - Converts dict → one big string (json.dumps) so we can split text.
   - Splits into chunks, wraps each in a Document with metadata (doc_id,
     resume_id, chunk_index, …).
   - Opens Chroma, deletes OLD chunks for this doc_id only, adds NEW chunks.
6) Result: vectors for that resume version are searchable; other versions
   stay as they are.


Why we json.dumps the dict after loading it
-------------------------------------------
load_parsed_resume returns Python objects (dict/list/str). The text splitter
only works on a string. dumps turns the dict back into JSON TEXT so we have
one string to split. The file on disk was already JSON; we reload as dict,
then stringify again on purpose for chunking.


The mistake that breaks everything
-----------------------------------
If the upload route saves to the DEFAULT file only:
  uploads/parsed_resume.json
but ingest_resume looks for:
  uploads/parsed_resumes/<resume_id>.json
then ingestion returns skipped / file not found. SAVE PATH and INGEST PATH
must match for the same resume_id.


What rag.py does NOT do
-----------------------
- It does not create resume_id (resume.py / client / DB does).
- It does not parse the PDF (resume_parser does).
- It does not replace your SQL “consultant ranking” (one embedding per whole
  resume vs JD). That is a separate feature. Chunk ingest is many embeddings
  per resume for retrieval/Q&A.


Files involved
--------------
- backend/ai/resume_parser.py — PDF → dict → save JSON (save_path parameter).
- backend/ai/rag.py — parsed_resume_json_path, _resume_doc_id, ingest_resume,
  get_vector_store, Chroma persist dir.
- backend/api/routes/resume.py — must pass resume_id and matching save_path
  when multi-resume is enabled (TODO if not done yet).


Environment
-----------
GEMINI_API_KEY required for embeddings. Python packages: see backend/
requirements.txt (langchain-chroma, chromadb, langchain-google-genai, …).


Optional next step (not ingestion)
----------------------------------
Add a function that runs similarity_search on Chroma for a question, filtered
by doc_id / resume_id, then calls the LLM — that is “RAG answer,” not ingest.
